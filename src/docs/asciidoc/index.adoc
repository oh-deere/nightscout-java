== Overview

Nightscout CGM server — Java 25 / Spring Boot 4 / PostgreSQL — implementing the upstream
Nightscout v1, v2, and v3 REST API contracts plus full LoopKit/Loop, AAPS, and xDrip+
client compatibility.

This document is generated from real Spring RestDocs tests against a running instance
backed by Testcontainers PostgreSQL. Every snippet shows the actual request and response
the service produced.

== Authentication

Two authentication methods are supported simultaneously. Either is sufficient.

=== API secret (Nightscout-native)

Set the `api-secret` HTTP header to the SHA-1 hash (lowercase hex) of the configured
`API_SECRET` environment variable. This is what xDrip+, AAPS, Loop, and the legacy
Nightscout web client use.

[source,bash]
----
HASH=$(echo -n "Jq6RH1NjytCPjnV" | shasum -a 1 | cut -d' ' -f1)
curl -H "api-secret: $HASH" https://nightscout.ohdeere.se/api/v1/entries.json
----

Alternatively, the access token may be passed as a query parameter:
`?token=<sha1-hash>`.

=== OAuth2 Bearer JWT (optional)

When `OAUTH_ENABLED=true` is set in the deployment environment, the service also accepts
JWTs issued by the configured OAuth2 issuer (`OAUTH_ISSUER_URI`, defaults to
`https://auth.ohdeere.se`). Send the token in the `Authorization` header:

[source,bash]
----
curl -H "Authorization: Bearer <jwt>" https://nightscout.ohdeere.se/api/v1/entries.json
----

JWT subjects map to Nightscout permissions through the `permissions` claim
(shiro-style strings like `api:entries:read`) or the standard OAuth `scope` claim. A
permission of `*:*:*` or `admin` grants full access.

=== Token exchange

Both auth methods can be exchanged for a short-lived JWT via:

* `GET /api/v2/authorization/request/{accessToken}`
* `GET /api/v3/authorization/request/{accessToken}` (alias)

The path variable can be a SHA-1 api-secret hash, an access token, or an existing JWT.
Returns `{token, sub, permissionGroups, iat, exp}`.

== Entries (CGM readings)

The highest-volume collection. Stores SGV, MBG, and calibration entries. Uploaders
deduplicate via `(sysTime, type)`.

=== `POST /api/v1/entries`

Accepts a single entry or an array. Returns the inserted records with server-assigned
`_id`, in the same order as the request, with HTTP 200.

== Treatments

Insulin doses, carb entries, temp basals, profile switches, site changes, and 25+ other
event types.

=== `POST /api/v1/treatments`

Accepts a single treatment or an array. Returns the inserted records with `_id`. The
optional `syncIdentifier` and `insulinType` fields are persisted and round-tripped
verbatim — required for Loop client reconciliation.
